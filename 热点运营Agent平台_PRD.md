# 热点运营Agent平台 - 产品需求文档（PRD）

> 版本：V1.0 | 日期：2026-05-17 | 状态：概念验证（POC）

---

## 一、产品概述

### 1.1 产品定位

**AI Agent驱动的热点运营一站式平台**——服务于抖音内容运营团队，覆盖热点线索感知→事件挖掘→运营执行→效果复盘的全生命周期。

**不是传统ERP后台，而是"运营的AI副驾驶"**。系统需要让运营同学感知到"有个智能体在帮我思考、推理、决策"，而非"一个自动填表工具"。

### 1.2 核心设计哲学

| 原则 | 含义 |
|------|------|
| **每个判断必须有"为什么"** | AI结论必须展示推理链路，不能只给结果 |
| **每个建议必须有来源** | RAG检索的SOP、历史案例、数据依据必须可见可查 |
| **Agent协作必须可视化** | 多Agent的分工和传递必须让用户感知到 |
| **决策必须有支撑** | 确认前必须看到效果预估、风险对比、相似案例 |
| **复盘必须能闭环** | 复盘数据→策略调优→A/B验证→知识库更新的链路完整可见 |

### 1.3 目标用户

**热点运营同学**：日常需要监测多平台热点、快速判断事件价值、制定运营策略、跟踪效果数据。核心痛点是"时间紧、信息杂、决策重"。

### 1.4 业务目标

| 指标 | 当前 | 目标（3个月） | 目标（6个月） |
|------|------|-------------|-------------|
| 热点发现到策略生成时效 | 30-60分钟 | 5分钟 | 2分钟 |
| 运营决策采纳率（Agent建议） | - | 60% | 75% |
| 策略命中率达预期效果比例 | - | 65% | 80% |
| 运营同学人均处理事件数 | 5-8个/天 | 15个/天 | 25个/天 |

## 二、系统架构

### 2.1 多Agent架构：分层集中式

采用**Orchestrator + 领域Agent**的分层集中式架构，而非平级流水线。

```
┌─────────────────────────────────────────────────────────┐
│                    人机协同层 (HITL)                     │
│            运营工作台 / 确认卡点 / 告警通道               │
└────────────────────────┬────────────────────────────────┘
                         │
                ┌────────▼────────┐
                │  Orchestrator   │
                │   中央调度Agent  │
                │                 │
                │ • 全局状态机管理 │
                │ • 任务动态编排   │
                │ • 人机卡点决策   │
                │ • 异常兜底       │
                │ • 上下文聚合路由 │
                └────┬───┬───┬───┘
                     │   │   │
        ┌────────────┤   │   ├────────────┐
        │            │   │   │            │
   ┌────▼──┐    ┌───▼──┐ ┌▼────┐    ┌───▼───┐
   │ 感知   │    │ 挖掘 │ │ 运营 │    │ 复盘  │
   │ Agent │    │Agent │ │Agent │    │ Agent │
   └──┬────┘    └──┬───┘ └─┬───┘    └───┬───┘
      │            │       │            │
   [信号采集]   [事件聚类] [SOP匹配]  [效果归因]
   [去噪排序]   [趋势预测] [策略生成]  [策略迭代]
   [初筛告警]   [影响评估] [动作编排]  [知识沉淀]
```

**为什么不用平级架构？**

热点运营有严格的业务流程和频繁的人工卡点，纯流水线无法处理：
- 挖掘Agent卡住时需要回调感知Agent补充线索
- 运营同学直接输入指令时可跳过感知Agent
- 复盘Agent发现偏差时需回退到运营Agent重新执行
- 任意环节都可能暂停等待人工确认

只有Orchestrator具备全局视角来做这些调度决策。

### 2.2 Agent职责定义

| Agent | 核心职责 | 输入 | 输出 | 触发方式 |
|-------|---------|------|------|---------|
| **Orchestrator** | 状态机管理、任务编排、人机卡点、异常兜底 | 全局事件状态+人工指令 | 下一步行动指令 | 持续运行/事件驱动 |
| **感知Agent** | 多源信号采集、去噪、初筛、优先级排序 | 多源数据流（搜索/视频/评论/外部API） | 结构化热点线索 | 流式触发（后台持续） |
| **挖掘Agent** | 事件聚类、实体抽取、生命周期判定、趋势预测 | 线索+历史事件图谱 | 结构化事件（含生命周期/影响评估） | Orchestrator调度 |
| **运营Agent** | SOP匹配（语义层+Skill）、策略生成、动作编排 | 事件+SOP库+历史案例 | 运营方案（脚本/规则/动作） | Orchestrator调度（需人工确认） |
| **复盘Agent** | 效果归因、策略评价、知识沉淀、自动调优建议 | 运营结果数据 | 复盘报告+策略调优参数 | 事件闭环后触发 |

**Agent间通信**：通过Orchestrator中转，消息格式标准化：
```json
{
  "event_id": "EVT_20240515_001",
  "from": "感知Agent",
  "to": "Orchestrator",
  "msg_type": "hot_clue_detected",
  "payload": { ... },
  "confidence": 0.92,
  "require_human_review": false,
  "timestamp": "2026-05-17T10:30:00Z"
}
```

### 2.3 RAG架构：Hybrid RAG（语义层+Skill + VectorRAG + GraphRAG）

不是"用语义层替代RAG"，而是**分场景使用最优方案**。

```
                         用户Query
                            │
                 ┌──────────▼──────────┐
                 │    Query Router     │
                 │    (意图识别+路由)   │
                 └────┬───────┬───────┘
                      │       │
            ┌─────────┘       └─────────┐
            │                           │
    ┌───────▼────────┐         ┌──────▼─────────┐
    │  结构化查询意图  │         │  开放理解意图    │
    │  "本月S级事件数"│         │  "这个事件怎么   │
    │  "热度增速Top10"│         │   看像之前的XX" │
    └───────┬────────┘         └─────┬───────────┘
            │                        │
    ┌───────▼────────┐         ┌────▼────────────┐
    │ 语义层 + Skill  │         │   RAG 层         │
    │                │         │                 │
    │ • 预定义原子指标│         │ • VectorRAG     │
    │ • 派生指标计算  │         │   （语义相似召回）│
    │ • 维度钻取     │         │ • GraphRAG      │
    │ • Skill调用    │         │   （关系遍历）    │
    │   (归因/预测)  │         │ • Query Rewrite │
    └───────────────┘         └─────────────────┘
```

**路由策略**：

| 查询类型 | 路由目标 | 原因 |
|---------|---------|------|
| 指标数值查询（热度/增速/ROI） | **语义层+Skill** | 指标预定义，亚秒级响应，口径统一 |
| SOP匹配、标准化策略 | **语义层+Skill** | SOP已封装为Skill，Function Calling直接调用 |
| 相似历史事件检索 | **VectorRAG+GraphRAG** | 需要语义相似+关系推理 |
| 开放域内容理解 | **VectorRAG** | 非结构化文本，需语义匹配 |
| 事件关联分析（多跳推理） | **GraphRAG** | 实体关系遍历，如"某明星→经纪公司→旗下艺人→关联事件" |
| 创意策略生成 | **LLM直连** | 超出预定义范围，需开放性推理 |

### 2.4 语义层设计

预定义指标和维度，形成"业务口径唯一真相源"。

**原子指标**：

| 指标名 | 定义 | 计算方式 | 维度 |
|-------|------|---------|------|
| 时效信号 | 内容新鲜度 | max(0, 28 - 距发布小时数) | 事件类型、来源平台 |
| 搜索增速 | 搜索量环比增速 | (近1h搜索量 - 上1h搜索量) / 上1h搜索量 | 关键词、地域 |
| 播放增速 | 播放量环比增速 | (近1h播放量 - 上1h播放量) / 上1h播放量 | 事件类型、生命周期 |
| 互动密度 | 互动率指标 | (评论+点赞+转发) / 播放量 | 内容类型、创作者层级 |
| 作者密度 | 参与创作者密度 | 参与创作者数 / 粉丝中位数 | 垂类、事件类型 |
| 情感得分 | 正向情感占比 | 正向评论数 / 总评论数 | 事件类型、时间窗口 |

**派生指标**：

| 指标名 | 公式 |
|-------|------|
| 热度总分 | Σ(wi × 原子指标i标准化值)，权重可配置 |
| 生命周期阶段 | f(热度增速, 持续时间, 情感得分, 作者密度) |
| 运营ROI | (增量播放量×CPM - 运营成本) / 运营成本 |
| 漏斗转化率 | 各环节转化数 / 上环节数 |

**Skill库**：

| Skill名 | 功能 | 输入 | 输出 |
|---------|------|------|------|
| `EventScoring` | 计算热度分数 | 事件ID + 原子指标值 | 热度总分 + 各因子贡献 |
| `LifecyclePredict` | 预测生命周期演化 | 事件时序数据 | 阶段预测 + 置信度 + 窗口建议 |
| `SOPMatch` | 匹配运营SOP | 事件画像（类型/生命周期/风险） | 匹配SOP + 置信度 + 推荐理由 |
| `Attribution` | 效果归因分析 | 事件ID + 运营动作 + 效果数据 | 各维度贡献度 + 因子拆解 |
| `AnomalyDetect` | 异常检测 | 指标时序数据 | 异常点 + 异常类型 + 根因推测 |
| `Forecast` | 趋势预测 | 历史热度数据 | 未来N小时预测 + 置信区间 |
| `ReportGenerate` | 复盘报告生成 | 完整事件数据 | 结构化复盘报告 |

### 2.5 召回策略：四级召回漏斗

```
Level 1: 元数据粗筛（毫秒级）
    基于时间范围、事件类型、生命周期的倒排索引
    10000 → 1000
         │
Level 2: 向量语义召回（10ms级）
    Embedding相似度（HNSW索引）
    1000 → 100
         │
Level 3: 图谱关系召回（50ms级）
    事件图谱2-3跳实体遍历
    补充20-30个多跳关联事件
         │
Level 4: Cross-Encoder精排（100ms级）
    轻量交叉编码器重排序
    130 → Top-K（5-10）
```

### 2.6 Chunk策略：文档类型感知

| 文档类型 | Chunk策略 | Chunk大小 | 原因 |
|---------|----------|----------|------|
| 运营SOP | 步骤级Semantic Chunk | 300-500 tokens | SOP步骤化，保持逻辑完整 |
| 历史事件报告 | 事件级Document-aware | 800-1200 tokens | 事件有完整叙事结构，不可切碎 |
| 政策法规 | 条款级Structure-aware | 200-400 tokens | 法条独立，需精确引用 |
| 舆情/评论 | 语义聚类Chunk | 动态 | 短文本先聚类再chunk |
| 视频内容 | 关键帧+ASR文本 | 按场景切换切分 | 多模态需保持视觉+文本一致性 |

## 三、功能模块设计

### 3.1 整体页面结构

```
┌──────────────────────────────────────────────────────────┐
│  Logo  │  工作台  |  事件详情  |  Agent编排  |  复盘指标   │
├────────┴─────────────────────────────────────────────────┤
│                                                          │
│                    主内容区（4个Tab）                      │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 3.2 Tab 1：工作台（事件看板）

**定位**：运营同学的"作战指挥室"，一眼看清全局态势，快速定位需要处理的事件。

**核心模块**：

#### 3.2.1 智能监控大屏

| 元素 | 说明 |
|------|------|
| 实时热点总数 | 按生命周期分色计数（萌芽蓝/爆发红/成熟黄/衰退灰） |
| 今日待处理/已处理 | 带环比箭头，红色警示待处理过多 |
| Agent状态指示器 | 脉冲动画区，显示各Agent实时状态 |
| 告警横幅 | P0事件到达、SLA超时、Agent异常等 |

#### 3.2.2 事件列表

每张事件卡片信息密度要高，一眼能判断优先级：

| 字段 | 展示方式 | 目的 |
|------|---------|------|
| 事件标题 | 单行文本，过长截断+tooltip | 快速识别事件 |
| 来源平台标签 | 彩色小标签（HN/微博/站内） | 判断来源可信度 |
| Sparkline迷你趋势 | 30px高折线图（近2小时） | 判断热度走势，上升/下降/平稳 |
| 生命周期 | 彩色圆点+文字（● 萌芽期） | 判断介入紧迫度 |
| Agent判断摘要 | 1行灰字（"模型发布类，建议2h内介入"） | AI初步结论 |
| SOP标签 | 可点击标签（SOP-TECH-001） | hover显示SOP摘要 |
| 风险等级 | 色块（绿/黄/红） | 一眼识别风险 |
| 置信度 | 0-100%颜色梯度 | 判断AI结论可靠度 |
| 操作按钮 | 查看详情/一键确认/加入待审 | 快速行动 |

**排序逻辑**：P0置顶（红色左边框高亮）→ 置信度降序 → 热度降序 → 时间倒序

**分组**：按事件类型自动分组（技术/娱乐/社会/体育），每组有组级统计。

### 3.3 Tab 2：事件详情（"单事件看清楚"）

**定位**：深度分析单个事件，让运营同学"看清楚"——不只是看数据，而是理解AI怎么分析的、为什么这样判断、该采取什么策略。

**布局**：左栏60%（事件分析）+ 右栏40%（Agent推理与推荐）

#### 3.3.1 左栏 - 事件动态分析区

**A. 热度态势（核心视觉区）**

| 元素 | 说明 |
|------|------|
| 主折线图 | X轴=时间（近24h+预测6h），Y轴=热度值。实线=历史，虚线=AI预测（带置信区间阴影），垂直虚线标"NOW" |
| 预测准确率 | 图例旁显示"历史预测准确率87%" |
| 右侧迷你面板 | S级标识+当前热度97+预测峰值103（2h后） |
| 因子拆解瀑布图 | 6个因子横向条形，展示各因子贡献了多少热度分。hover显示详情 |

**B. 生命周期阶段**

| 元素 | 说明 |
|------|------|
| 横向流程图 | 萌芽期→爆发期→成熟期→衰退期，当前阶段脉冲动画高亮 |
| 预测面板 | "预计6-12h后进入爆发期"+置信度85%+"建议介入窗口：未来2h内" |
| 历史对比 | "同类事件平均萌芽→爆发8h，当前进度偏快" |

**C. 热度因子与运营判断**

7个因子逐行展示，每行包含：因子名、进度条（当前/满分）、状态标签（优秀/良好/需关注）、AI解读、同类对比。

因子列表：时效信号、传播速度、作者密度、情感倾向、内容质量、平台适配度、风险因子。

**D. 来源与摘要**

| 元素 | 说明 |
|------|------|
| 来源可信度评分 | 1-5星+来源类型标签 |
| 多源时间线 | 各来源报道差异对比 |
| 原始链接 | 快速跳转 |

#### 3.3.2 右栏 - Agent推理与推荐区

**A. Agent推理链路可视化（必须）**

展示Agent分析该事件的完整链路：

```
感知Agent → 挖掘Agent → 策略Agent
[发现线索]  [事件聚类]  [SOP匹配]
 23:25       23:26       23:27
置信92%     置信88%     置信85%
```

- 三节点横向排列，箭头连接
- 每节点hover展开：Agent做了什么、用了什么数据源、输出什么结论
- 当前活跃步骤有脉冲动画

**B. 相似历史事件推荐（RAG核心体现）**

- 标题："相似历史事件（基于知识图谱+向量召回）"
- 2-3个事件卡片，每张含：事件名、相似度%、当时生命周期/策略/效果
- 底部总结："基于3个相似案例，推荐'技术解读+二创激励'策略"

**C. 推荐策略卡片**

| 元素 | 说明 |
|------|------|
| 策略名称 | 如"技术类模型发布-萌芽期-低风险" |
| 策略来源 | 可点击标签（匹配SOP-TECH-001），点击查看SOP原文 |
| 预期效果 | 迷你柱状图（播放量+130%、互动率+15%） |
| 风险提示 | "该策略在娱乐类事件中效果下降40%，请确认分类准确" |
| 关键动作清单 | ①②③编号列表 |
| 操作按钮 | 采纳策略并生成方案 / 查看其他策略 / 人工调整 |

### 3.4 Tab 3：Agent编排（"人工确认台与动作清单"）

**定位**：运营方案的编辑、确认、执行控制中心。核心设计准则是"人机协同"——Agent生成初稿，运营同学审核确认，系统追踪执行。

**布局**：左栏50%（策略编辑）+ 右栏50%（执行预览与确认）

#### 3.4.1 策略编辑区

**A. 结构化意图编辑器**

| 元素 | 说明 |
|------|------|
| 自然语言输入框 | 保留，但下方实时展示意图解析结果 |
| 意图标签卡片 | [意图：风险评估] [对象：当前事件] [输出：抖音脚本+评论区] |
| 手动修正 | 标签可点击修正 |
| 快捷指令按钮 | "生成快反脚本"/"评估风险"/"生成评论话术"/"制定放量策略"/"全链路自动运营" |

**B. 短视频脚本 - 分镜时间轴**

- 横向时间轴，每3秒一个分镜卡片
- 每个分镜含：时间戳、画面描述（可编辑）、字幕文案（高亮）、BGM情绪、镜头语言
- 卡片可拖拽排序
- 底部：总时长计数器+预估完播率
- "AI优化"按钮：可选优化方向

**C. 评论区承接 - 评论运营矩阵**

| 元素 | 说明 |
|------|------|
| 置顶评论策略 | 1-3条预设置顶评论，每条可编辑，显示预期效果 |

#### 3.5.1 顶部指标区

5个指标卡片，每个含：数值、环比变化（↑↓）、迷你Sparkline（7天趋势）、目标达成进度条。

#### 3.5.2 决策漏斗

- 桑基图/漏斗图：事件发现→Agent分析→方案生成→运营确认→执行完成→效果达标

## 四、人机协同（HITL）设计

### 4.1 人机边界定义

| 环节 | 机器职责 | 人工职责 | 协作模式 |
|------|---------|---------|---------|
| 热点感知 | 自动采集、初筛、去噪 | 设定信源优先级和过滤规则 | 机器自动，人配置 |
| 事件挖掘 | 自动聚类、生命周期判定 | 修正事件分类和关键实体 | 机器建议，人确认 |
| 策略匹配 | 自动匹配SOP、生成方案 | 审核方案、创造性调整 | 机器生成，人决策 |
| 运营执行 | 自动执行低风险动作（如定时发布） | 高风险动作人工确认 | 分级自动+人工确认 |
| 风险审核 | 自动风控扫描、敏感词检测 | 最终风险评估和处置决策 | 机器预警，人决策 |
| 效果复盘 | 自动归因分析、生成报告 | 定性判断、策略方向决策 | 机器分析，人总结 |

### 4.2 卡点设计

| 卡点位置 | 触发条件 | 等待时长 | 超时策略 |
|---------|---------|---------|---------|
| 策略确认 | Agent生成方案后 | SLA 10分钟 | 自动降级为低风险方案执行 |
| 风险审核 | 风险等级≥中 | SLA 15分钟 | 自动限流，等待人工 |
| 改写确认 | 运营点击"改写"后 | SLA 20分钟 | 保留原方案，通知提醒 |
| 复盘确认 | Agent生成复盘后 | SLA 30分钟 | 自动归档，标记"待补充" |

### 4.3 干预机制

- **全局否决**：运营可在任意环节点击"否决"，终止当前事件处理流程
- **状态回退**：复盘发现问题可回退到运营Agent重新生成策略
- **参数实时调整**：运营可调整热度因子权重、阈值等参数，立即生效
- **Agent行为反馈**：每条Agent输出旁有"👍有用/👎无用"反馈按钮，数据回流到模型优化

## 八、迭代路径

### Phase 1（1-2周）：MVP验证
- 单Agent + VectorRAG，聚焦"事件→SOP匹配→策略生成"一个闭环
- 核心指标：SOP匹配准确率>85%

### Phase 2（2-4周）：多Agent协作
- 感知+挖掘+运营三Agent协作
- 引入GraphRAG做事件关联
- 核心指标：运营采纳率>60%

### Phase 3（4-8周）：智能化闭环
- 复盘Agent + 效果归因 + 自动调优
- 引入语义层+Skill做结构化查询
- 核心指标：策略命中率>75%

### Phase 4（8-12周）：规模化
- A/B测试框架 + 策略自动迭代
- 多事件并行处理 + 批量运营
- 核心指标：人均处理事件数提升3x
